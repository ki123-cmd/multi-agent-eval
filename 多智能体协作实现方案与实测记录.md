## 零、先回答：能不能用 openspec？

**能，且已经用上了。但只用在 harness 实现层。**

`openspec init --tools claude,codex` 会**对称地**生成 `.claude/skills/` 与 `.codex/skills/`（各 5 个 skill）。这正好命中原方案那条硬性不变量——「`task.md` 作为所有 agent 共享的唯一需求来源」。openspec 的 spec 目录天然就是这个唯一真源，比手工维护 `task.md` 更不容易漂移。**[已验证]**

**不用在实验层。** openspec 管的是「变更提案 → 规格 → 归档」，适合描述「要造什么」。本项目的最终产物是评估报告，报告不是 spec，硬塞进 change 流程只会变成仪式开销。

实际落地：`openspec/changes/cross-review-harness/`，含 `proposal.md` / `design.md` / `tasks.md` / 4 份 `specs/`，`openspec validate` 通过。

> **前置修复 [已验证]**：`D:\Cross-model-review\.git` 原本是个**空目录**（0 个文件），`git status` 报 not a repository。原方案第 2 步「产出当前变更 diff」和案例 A 的 `git diff 提取变更` 全都依赖 git，实际跑不起来。已 `git init` 重建。

## 一、被推翻的三个假设

这一节最重要。原方案的三个核心判断经实测/核实后不成立，如果直接照原方案实施，产出的数据会全部作废。

### 假设 1（推翻）：「Windows 沙箱不可用是个可绕过的 bug」

原文档把它当成 push 模式的一个理由。实测表明它比这更严重：

| sandbox 模式 | Codex 能否读文件 | 现象 |
|---|---|---|
| `read-only` | ❌ | `CreateProcessWithLogonW failed: 1385` |
| `workspace-write` | ❌ | 同上 |
| `danger-full-access` | ✅ | 正常读出 |

**[已验证]** 坏的是**沙箱机制本身**，不是某个级别。要让 Codex 自己读文件，就必须**关掉沙箱**——而评审者恰恰必须只读。二者不可兼得。

**推论（本文新增）**：任何依赖「Codex 自主读仓库」的路径在本机都不可用，**包括官方插件 `openai/codex-plugin-cc`**（其 `/codex:review`、`/codex:rescue`、`/codex:transfer` 均依赖 Codex 的本地环境访问）和内置的 `codex exec review`。

调研 agent 曾建议「直接复用 codex-plugin-cc，这是最成熟的一手方案」。**在这台机器上这个建议是错的**，而错因只有实测能发现。

> **所以：push 上下文在本机不是性能优化，是唯一正确解。**

### 假设 2（推翻）：「schema 通过 + 退出码 0 = 评审成功」

原方案的判定逻辑依赖这两个信号。实测证明它们在静默故障下**同时成立**。

让 Codex 以 pull 模式评审一个真实 PR（`sentry-greptile#1`），结果：

```
codex EXIT CODE = 0
sandbox 1385 error in stderr: True
```
```json
{ "canary": {"changed_file_count": 0, "first_changed_file": ""},
  "confidence": "low", "critical_issues": [], "suggestions": [],
  "verdict": "FAIL" }
```

退出码 0、schema 合规、`verdict: "FAIL"`。跑在流水线里，你会记录下「Codex 判定 FAIL」，而真实原因是**它一个字节的代码都没读到**。**[已验证]**

**对策（本文新增 · canary 字段）**：verdict 必须回报只有真读了 diff 才能得出的事实——
- `first_changed_file`：首个 `diff --git` 行的 `b/` 路径
- `changed_file_count`：`diff --git` 行的总数

对不上就判 `INVALID`，不计入任何统计。**这把静默故障从沉默变成硬失败。**

harness 实跑对上述假 FAIL 的裁定：

```
>>> HARNESS RULING: INVALID
   - canary.first_changed_file='' expected 'src/sentry/api/endpoints/organization_auditlogs.py'
     -- reviewer did not see the diff
   - canary.changed_file_count=0 expected 3 -- reviewer did not see the diff
   - verdict=FAIL contradicts 0 critical_issues
```

### 假设 3（推翻）：「20-30 条样本，规律就会浮出来」

**[已核实]** 配对 McNemar 设计下，要在 α=0.05、80% power 检出 15–20pp 的真阳性率差异（3 组两两比较需 Bonferroni 校正到 α=0.0167）：

| 目标效应 δ | 不一致率 p_disc | 所需 golden item 数 |
|---|---|---|
| 20pp | 0.30 | ~76 |
| 20pp | 0.40 | ~102 |
| 15pp | 0.30 | ~137 |

**20–30 条只够做试点去估计 `p_disc`，不足以支撑任何定量结论。** 正式实验需要 **55–140 条**。

而且两个 CLI **都不暴露 `--temperature` / `--seed`` **[已验证]**，无法固定种子，方差只能靠重复采样控制。依据 ICC 收敛研究，结构化任务需 8–16 次重复，复杂推理需 ≥32 次。

## 二、证据基础：哪些是真的

原方案标了两条 `[未证实]` 的 arXiv 引用，我一度强烈怀疑是幻觉。**用一篇已知真实的论文（1706.03762）作控制组核实后，两篇都是真的。** 这个控制组很关键——我第一次用 `curl` 查 arXiv 时三个 ID 全部返回空，若无控制组就会误判「论文不存在」，实际是网络链路不通。

| 来源 | 状态 | 与本项目的关系 |
|---|---|---|
| arXiv 2603.12123《Cross-Context Review》 | **[已核实存在]** | **最重要的一篇**。同一模型的四组对照：SR（同会话自审）F1 24.6%、SR2（重复自审）21.7%、SA（带上下文子代理）23.8%、**CCR（隔离会话）28.6%**。结论：**收益来自上下文隔离，不是模型多样性**；重复自审甚至更差。**它没有跨模型臂。** |
| arXiv 2603.03406《Review Beats Planning》 | **[已核实存在]** | 双模型，但研究轴是「规划 vs 评审」角色分配，非「跨模型 vs 同模型隔离」。 |
| arXiv 2404.13076 | [已核实] | LLM 判官偏爱自己的生成，且自我识别能力与偏好强度线性相关。 |
| arXiv 2410.21819 | [已核实] | 机制修正：偏好实为**低困惑度/风格熟悉度**，未必是字面自我识别。 |
| arXiv 2605.29800 | [已核实] | 9 模型判官团只提供约 **2 票**独立信息；单个好判官即可匹敌。 |
| Anthropic 多智能体研究系统 | [已核实] | 隔离是刻意设计（减少路径依赖），代价 **~15× token**。 |
| Cognition《Don't Build Multi-Agents》及其修正 | [已核实] | 后续修正为条件性立场：**只有一个 agent 写**、评审者拿**干净上下文**、双方都得是前沿模型。 |
| 「跨模型多抓 40–60% 问题」 | **[未证实]** | 溯源全部指向无方法论的内容营销。 |

> **关键空白**：**没有人公开跑过「跨模型隔离 vs 同模型隔离」的对照实验。** CCR 论文做了同模型的隔离对照但无跨模型臂；santa-loop / Star Chamber 做了跨模型但无同模型隔离基线。这正是本项目的价值所在——也意味着我们不能引用任何现成数字。

## 三、复用了什么，自建了什么

原则：**先检索，再自建。** 第 3 点要求「学会复用别人已经完成的方案」，以下每一项都经直接访问核实。

| 资产 | 许可 | 决定 | 理由 |
|---|---|---|---|
| **`withmartian/code-review-benchmark`** | MIT | **直接复用真值集** | 5 个真实仓库（sentry/grafana/keycloak/discourse/cal.com）50 个 PR、**136 条 golden comments**，无需 Docker。136 恰好落在 55–140 的功效区间内。**[已验证：已 clone 并统计]** |
| `openai/codex-plugin-cc` | Apache-2.0 | **不用** | 依赖 Codex 本地读仓库 → 撞上假设 1。**在本机不可用。** |
| `codex exec review` 子命令 | — | **不用** | 同上。 |
| Mozilla.ai `star-chamber` | Apache-2.0 | 设计参考 | 明确自我定位为 **advisory 而非 blocking**，与「区分 critical/suggestions」一致。 |
| `formin/multi-model-review` | MIT | 设计参考 | 文件化交接 + 置信度门控自动应用；6 star 单人维护，不作依赖。 |
| santa-loop / santa-method | 无稳定包 | 协议参考 | 「双独立评审 + 无共享上下文 + 每轮换全新评审者」。其三个缺陷（静默降级、`-m gpt-5.4` 非最强、双 PASS 后自动 `git push`）须先修。 |
| SWE-bench-Live | MIT | 备选 | 月度刷新真实 issue，抗污染；但**需 Docker**。留给 bug-fix 场景。 |
| FEA-Bench | MIT | 备选（有缺陷） | 唯一对口的**功能实现**基准，但数据冻结于 2025-03，**污染风险高**。**不存在截止日后的功能实现基准。** |

**自建的只有三样**（因为确实没有现成的）：
1. **canary 校验** —— 没有任何现成 harness 做这件事，而本机不做就没法拿到可信数据
2. **跨模型 vs 同模型隔离的对照编排** —— 公开研究里的空白
3. **对称适配器** —— 抹平 `--output-schema` 的能力不对称

## 四、CC 与 Codex 的协作方式

### 4.1 三种连接方式的定位

它们不是三条并列路线，而是三个抽象层级。**真正的研究变量不是连接方式本身，而是它解锁的协作形状。**

| | 能力天花板 | 确定性 | 会话延续 | 上下文隔离 | 本机可用性 |
|---|---|---|---|---|---|
| Bash shell-out | 完整 CLI 参数面 | 脚本写死 | ✗ | ✓ 进程级 | ✅ |
| `codex mcp-server` | 固定 tool schema | 模型自主决定 | ✓ | ✗ 结果回流主上下文 | 未验证 |
| `codex-plugin-cc` | 官方封装命令集 | 内部不透明 | 部分 | 视命令而定 | ❌ 撞假设 1 |

### 4.2 环节 × 连接方式矩阵（本文新增）

原方案把「连接方式」和「协作场景」一一对应了（互查↔shell-out、互补↔MCP、互接↔插件），这是**过度简化**。单个「互查 Review」场景内部就有 7 个环节，每个环节的最优通道不同。

| 环节 | 推荐通道 | 理由 | 置信度 |
|---|---|---|---|
| 1. 任务输入 / spec 定义 | **文件产物**（openspec spec） | spec-kit、Kiro 两大工具独立收敛到「文件作为阶段边界 + 人工闸门」 | 中高（模式），无因果证明 |
| 2. Agent A 实现 | **shell-out**；禁止其自行开 MCP 咨询 | MCP 规范 SEP-2575 自陈「状态化是最后手段」；模型自主决定调用时机会破坏可复现性 | 中 |
| 3. 构造 review package | **文件包 + sha256 锚定** | 双评审必须看到逐字节相同输入，否则重合率无法归因 | 高（本项目硬需求） |
| 4. Agent B 独立评审 | **shell-out，全新进程，push 上下文** | **本矩阵证据最强的一格**：Anthropic「隔离减少路径依赖」+ 自我偏好偏差文献双向印证 | **高** |
| 5. 判定 PASS/FAIL | **shell-out + `--output-schema`** | 约束解码显著优于自由文本解析（JSONSchemaBench） | 高（方向） |
| 6. Agent A 修复 | 需来回澄清才用 MCP；否则重发 verdict 文件即可 | `codex-reply()` 官方定位为「迭代精修」；单次修复用不上状态化 | 中（判断题，无直接研究） |
| 7. 记录指标 | **文件/日志持久化**，不接模型 | — | **无证据，须自行测量** |

> 第 4 格是唯一有两条独立文献同向支撑的。第 7 格没有任何公开指导。**其余格子都是跨领域综合推断，不是直接引用。**

### 4.3 六个协作场景的推进顺序

| 场景 | 本版是否覆盖 | 依据 |
|---|---|---|
| **互查 Review** | ✅ 主体 | 可判定性最强，成本最低 |
| **同模型隔离对照** | ✅ 主体 | 唯一能证伪「跨模型有增量」的设计（原方案坑 4） |
| 并行评审 Consensus | ⏳ 次轮 | 需先有单评审基线 |
| 互补 Consult（MCP） | ⏳ 后置 | 必须先知道 Codex 强在哪；且 MCP 会污染观测（坑 1） |
| 互接 Handoff | ⏳ 后置 | — |
| 主从式 Orchestrator-Worker | ⏳ 后置 | Cognition：只有一个 agent 能写 |

**任务类型**不止 bug-fix。但**截止日后的功能实现基准不存在**，功能实现场景的真值集需另行设计（见第七节）。

## 五、模型参数配置

### 5.1 实测可用模型 **[已验证 `codex debug models`]**

```
gpt-5.5          ← frontier，评审主力
gpt-5.4
gpt-5.4-mini     ← 冒烟测试
gpt-5.3-codex
gpt-5.2
codex-auto-review
```

> ⚠️ santa-loop 里写死的 `-m gpt-5.4` 是有效模型名，但已非最强。

### 5.2 评审者调用参数（对称设计）

**Codex 侧**
```bash
codex exec \
  --sandbox read-only \        # 硬约束；本机下它读不到文件，正因如此必须 push
  --skip-git-repo-check \
  --ephemeral \                # 不持久化会话
  -m gpt-5.5 \
  -c model_reasoning_effort=medium \
  --output-schema harness/schema/verdict.schema.json \
  -o result.json \             # 结果提取用这个，不要解析 --json 事件流
  - < review-package.txt       # push 上下文
```

**Claude 侧**
```bash
claude -p \
  --model claude-opus-4-8 \
  --output-format json \
  --disallowedTools "Bash,Read,Glob,Grep,Edit,Write,WebFetch,WebSearch,Task,NotebookEdit" \
  --max-turns 1 \
  < review-package.txt         # 同一个文件，同一个 sha256
```

### 5.3 三处必须记录的不对称

| 维度 | Codex | Claude | 处理 |
|---|---|---|---|
| 结构化输出强制 | `--output-schema` 原生支持 | **不支持** | schema 内联进 prompt（两侧共享，保持字节一致）+ 对称的一次修复重试；记录 `schema_failure_rate` |
| 推理档位 | `model_reasoning_effort`：low/medium/high/xhigh | `--effort` | 跨臂固定，不作隐藏变量 |
| 固定开销 | 一次 trivial 调用 ~4,400 token | 一次 trivial 调用 **$0.228**，含 ~18,100 cache-creation token 的系统提示 | 「每真阳性成本」必须扣除固定开销后再比 **[已验证]** |

### 5.4 裁判模型

`ollama` 本地 `gpt-oss:120b-cloud`，`temperature=0`，`format=json`。

**为什么不能用 Claude 或 Codex 当裁判**：arXiv 2404.13076 / 2410.21819——判官偏爱自己（及风格熟悉）的生成，会直接污染我们唯一关心的那个对比。

**残余风险（必须写在报告里）**：gpt-oss 仍是 OpenAI 血统，对 Codex 的措辞可能有熟悉度偏好。缓解：人工抽检 10–20% 并报告 Cohen's κ。**不采用判官团**——arXiv 2605.29800 表明 9 判官只值约 2 票独立信息。

## 六、实验设计

### 6.1 臂的定义

原方案的三臂设计（跨模型 / 同模型隔离 / 同模型共享）**不足以归因**。若 Codex 本就比 Claude 更擅长评审，「跨模型更强」这个结论就是假的——它测的是「Codex > Claude」，不是「多样性 > 同质性」。

**必须补镜像臂**：

| 臂 | 实现者 | 评审者 | 上下文 | 分离的变量 |
|---|---|---|---|---|
| A1 | Claude | Codex | 隔离 | 跨模型 + 隔离 |
| A2 | Claude | 全新 Claude | 隔离 | 同模型 + 隔离（CCR 复现） |
| A3 | Claude | Claude 自审 | 共享 | 基线 |
| **A4** | **Codex** | **全新 Codex** | **隔离** | **镜像：区分「多样性」与「某模型本就更强」** |
| A5 | Codex | Claude | 隔离 | 镜像跨向 |
| A6 | Codex | Codex 自审 | 共享 | 镜像基线 |

> **注意一个设计裂缝**：在 CRB 这类**固定 golden PR 集**上，没有「实现者」——代码是人写的。因此 **A3/A6 自审臂在该场景下无定义**。上下文隔离维度必须靠独立的「先实现后评审」实验族（注入缺陷）来测。本版先跑评审能力与重合率，隔离维度留给第二族。

### 6.2 指标

- **召回率** = 命中的 golden comment / 全部 golden comment
- **精确率** = 命中的 finding / 全部 finding（`1 − 精确率` 即噪音率）
- **重合率**（Jaccard，对命中的 golden 取交并）—— **判断跨模型是否真有增量的核心指标**
  - 高重合 → 多样性没带来增量，钱白花
  - 低重合 + 高真阳性 → 真正互补
  - 低重合 + 低真阳性 → rubric 太松
- **每真阳性成本** = 扣除固定开销后的 $ / 真阳性数
- `INVALID 率`、`schema_failure_rate`、`repair 次数` —— harness 健康度，不得掩盖

### 6.3 反作弊约束

- **绝不静默降级**：某后端缺失 → 硬失败退出，绝不用另一个模型冒充该臂（santa-loop 坑 2）
- **绝不静默截断**：diff 超预算 → 该 item 记入 `skipped`
- **裁判宕机 ≠ 零召回**：裁判失败 → `JUDGE_ERROR`，排除出统计
- **输入字节一致**：同 item 所有臂共享同一 `sha256`

## 七、实测效果

> **这是一次试点（pilot），不是结论。** 样本量 4 个 item / 12 条 golden comment，远低于第一节算出的 55–140。它的用途是：验证 harness 可跑通、估计不一致率 `p_disc`、暴露设计缺陷。**任何人不得引用下表数字作为「Claude 比 Codex 强」的依据。**

### 7.1 场景与配置

| 项 | 值 |
|---|---|
| 场景 | 互查 Review（真实 PR 的代码评审） |
| 真值集 | `withmartian/code-review-benchmark`，sentry 前 4 个 PR，共 12 条 golden comment |
| 评审输入 | push 上下文，sha256 锚定，两臂逐字节相同 |
| Codex 臂 | `gpt-5.5`，`model_reasoning_effort=medium`，`--sandbox read-only --ephemeral --output-schema` |
| Claude 臂 | `claude-opus-4-8`，全部工具禁用，`--max-turns 1` |
| 裁判 | `gpt-oss:120b-cloud`（ollama，`temperature=0`），第三方 |
| 重复次数 | **1**（不足，见边界） |

### 7.2 结果

```
arm               valid  invalid  repaired   recall  precision   med_ms
----------------------------------------------------------------------
claude-opus           4        0         1    33.3%      40.0%   105076
codex-gpt5.5          4        0         0    16.7%      40.0%    20386
----------------------------------------------------------------------
overlap (Jaccard over matched golden): 2/4 = 50.0%
```

| 指标 | Codex `gpt-5.5` | Claude `opus-4-8` |
|---|---|---|
| 召回率 | 2/12 = **16.7%** | 4/12 = **33.3%** |
| 精确率 | 2/5 = 40.0% | 4/10 = 40.0% |
| 中位耗时 | **20.4 s** | 105.1 s（慢 5.2×） |
| 总成本 | **未采集**（harness 缺口，见下） | $1.541 |
| 每真阳性成本 | 未采集 | $0.385 |
| 需要修复重试 | 0/4 | 1/4 |

### 7.3 逐条重合分析

| item | Codex 命中 | Claude 命中 | 双方共识 | Codex 独家 | Claude 独家 |
|---|---|---|---|---|---|
| sentry-0 | [0] | [0] | [0] | — | — |
| sentry-1 | [0] | [0] | [0] | — | — |
| sentry-2 | — | [1] | — | — | [1] |
| sentry-3 | — | [2] | — | — | [2] |

**这张表比召回率更重要。**

重合率 50% 看起来像「跨模型有互补」，但拆开看，**互补是单向的**：Codex 在这 4 个 item 上**没有找到任何一条 Claude 没找到的问题**。

按原方案第七节的解读框架，这落在「**高重合 → 模型多样性没带来增量**」与「Codex 纯粹是更弱的评审者」之间，而 4 个 item 无法区分二者。**这恰恰是第六节要补 A4 镜像臂的原因**：不知道 Codex 评审自己写的代码表现如何，就无法判断它是「视角不同」还是「就是更弱」。

### 7.4 试点暴露的三个缺陷（均已修或已记录）

**① schema 不对称差点制造出彻底错误的结论。**

第一次跑，Claude 臂 4/4 全部 `INVALID`，表面数据是「Claude 召回率 0%」。查原始输出才发现：Claude 的 `canary` **完全正确**，它读到了 diff，并且找到了 sentry-0 那条 High 级 golden bug——

> `Django QuerySet slicing does not support negative indexing`

它被判 0 分，只因为它自己发明了字段名（`location` / `issue`），漏了 `confidence`。**Codex 有 `--output-schema` 强制约束，`claude -p` 没有。**

如果不查这一步，报告会写成「Codex 22% vs Claude 0%」——而真相是 Claude 找到了 Codex 漏掉的那条 bug。**这是本次试点最有价值的一个发现：能力面不对称会伪装成能力差异。**

修法：schema 内联进 prompt（两侧共享，sha256 不变）+ 两侧对称的一次修复重试。修复后 8/8 全部 VALID。

**② 同一份输入，两次运行结论不同。**

`sentry-0`，Claude 臂，`sha256=95616c2c0bd4`（**逐字节相同**）：

| 运行 | findings | 命中 |
|---|---|---|
| pilot-2 | 5 | **3/3** |
| pilot-3 | 2 | **1/3** |

两个 CLI 都不暴露 `seed` / `temperature`，无法固定。**单次运行的召回率是不可复现的。** 这直接实证了「每 item 需重复 8–16 次」的要求——而本试点只跑了 1 次，所以 7.2 那张表的每个数字都带着未测量的方差。

**③ Codex 侧成本未被采集。**

`-o` 只落最终消息，不含 token 用量；用量在 `--json` 事件流里。当前 harness 因此**无法计算 Codex 的每真阳性成本**，表中如实留空而非估算。已记入待办。

### 7.5 试点给出的唯一可用结论

- harness 可跑通，canary 能拦住静默故障（负向测试已验证）
- **不一致率 `p_disc` 初估 ≈ 2/4 = 0.5**（两臂在 4 个 item 上有 2 个 item 结论不同）。若 `p_disc≈0.4–0.5`，按第一节的功效公式，正式实验需 **~102–140 个 golden item** —— CRB 的 136 条**刚好够用，且没有余量**
- Codex 快 5×，Claude 召回高一倍。两者精确率相同（40%），即**噪音水平相当**

## 八、诚实的边界

1. **样本量不足以下定量结论。** 本文的试点只是用来估计 `p_disc` 和验证 harness，不是结论。
2. **golden set 有记忆污染风险。** CRB 的 50 个 PR 早于模型知识截止（2026-01）。缓解方案是用其 online 模式思路抽取截止日后的新 PR 作第二批，对比两批 TPR 差异。**尚未做。**
3. **裁判是单个第三方模型**，且与 Codex 同源。人工抽检与 κ **尚未做**。
4. **A3/A6 自审臂在本场景无定义**，上下文隔离维度尚未真正测到。
5. **重复采样次数为 1**，远低于 ICC 收敛所需的 8–16 次。
6. **MCP 通道完全未验证**（`codex mcp-server` 未实机跑过）。
7. **功能实现场景没有可信真值集**，这是公开生态的空白，不是本项目的疏漏。

> 要在团队推这件事，唯一诚实的做法是自己测，而不是引用别人的数字——包括不要引用本文这份试点数据。
