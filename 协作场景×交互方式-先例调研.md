## 一、先纠正三处：原文档与事实不符

这三条都会直接改变实验设计，必须先摆平。

### 1. shell-out **能**多轮会话 —— 原对比表写的 ✗ 是错的

原《ClaudeCode与Codex交叉使用方案》断言：「`codex mcp-server` 是三种方式里**唯一**能跨多次调用保持 Codex 会话状态的」。

本机实测 **[已验证]**：

```bash
# 进程 1
$ echo "Remember this codeword: PURPLE_OTTER_42. Reply only: OK" \
    | codex exec --sandbox read-only --skip-git-repo-check -m gpt-5.4-mini -o r1.txt -
OK

# 进程 2 —— 全新进程
$ echo "What codeword did I ask you to remember?" \
    | codex exec --skip-git-repo-check --sandbox read-only resume --last -o r2.txt -
PURPLE_OTTER_42
```

`codex exec resume --last` / `resume <SESSION_ID>` 在**独立进程**中恢复了会话。开源项目 `AltimateAI/claude-consensus` 正是用这条做多轮收敛 **[已核实源码]**：

```bash
codex exec resume --last - < $SESSION_DIR/convergence-prompt-{MODEL_ID}.md
```

> **影响**：MCP 通道**失去了它唯一的独占卖点**。「需要多轮会话所以必须上 MCP」这个论证不成立。

（注意：`--ephemeral` 会阻止会话落盘，故 resume 与 `--ephemeral` 互斥。）

### 2. 官方插件走的是 `codex app-server`，既不是 `codex exec`，也不是 MCP

原文档说插件「调用本机 codex 二进制」，把它和 shell-out 混为一谈。**源码核实后是第三种东西** **[已核实源码]**：

`plugins/codex/scripts/lib/app-server.mjs`
```javascript
this.proc = spawn("codex", ["app-server"], { stdio: ["pipe","pipe","pipe"], ... });
await this.request("initialize", {...});
```

行分隔 JSON over stdin/stdout，调用 `thread/start`、`review/start`、`turn/start`、`externalAgentConfig/import` 等 RPC。**全仓库没有任何 `codex exec` 调用，也没有 MCP client/server 角色。**

所以三种交互方式实际上是 **四**种传输层：`codex exec` shell-out、`codex app-server` JSON-RPC、`codex mcp-server` MCP、以及反向的 `claude -p`。

### 3. santa-loop 是 push 模式 —— 原文档说它「依赖 Codex 自主读文件」是错的

本机 `ecs/0.8.14/commands/santa-loop.md` **[已验证]**：

```bash
PROMPT_FILE=$(mktemp ...)
cat > "$PROMPT_FILE" << 'EOF'
... full rubric + file contents + reviewer instructions ...
EOF
codex exec --sandbox read-only -m gpt-5.4 -C "$(pwd)" - < "$PROMPT_FILE"
```

**文件内容是内联进 prompt 的**，走 stdin。所以它不会撞上「坑 0」的静默假 FAIL。

它真正的缺陷是另外三个（均已核实）：
- **无 `--output-schema`** —— 靠自由文本解析 JSON
- **静默降级** —— 检测不到 `codex`/`gemini` 时 fallback 成第二个 Claude Opus，只留一行警告
- **双 PASS 后 `git push -u origin HEAD`** —— 自动推送
- 附带：`-m gpt-5.4` 已非最强；仍传 `-C "$(pwd)"`，Codex 可能仍尝试工具调用

## 二、六个场景的理论依据

**关键结论：六个场景的证据强度差了整整三个量级。** 不能用同一套说辞给它们背书。

| # | 场景 | 权威命名 | 是否有效果测量 | 判定 |
|---|---|---|---|---|
| 1 | **互查 Review** | Anthropic **Evaluator-Optimizer**；Google ADK **Generator-and-Critic** | ❌ 仅架构描述，无「独立评审 + 阻塞门禁」的对照基准 | **成立的命名模式** |
| 2 | **互补 Consult** | 无任何厂商/学术命名 | ❌ | **[弱/传闻]** 最近亲 MoA（arXiv 2406.04692）是**预设的多层并行集成**，不是「卡住了临时问一句」 |
| 3 | **互接 Handoff** | OpenAI Agents SDK `handoff()`；LangGraph `Command` | ❌ 两个 SDK 默认都是**传整份会话历史**，不是文件 | **原语成立**；但「文件交接优于共享会话」**无任何对照证据** |
| 4 | **并行评审 Consensus** | PoLL（arXiv 2404.18796）；Self-Consistency（2203.11171） | ✅ **有独立测量** | **成立且有数据**，但见下方反证 |
| 5 | **测试资产评审** | **无** | ❌ | **本地原创，无先例** |
| 6 | **主从式 Orchestrator-Worker** | Anthropic **Orchestrator-Workers**；Google **Hierarchical Decomposition**；CrewAI **Hierarchical Process** | ✅ 厂商自测：+90.2% 质量、-90% 时延 | **证据最强**，但见下方反证 |

### 必须一并记录的三条反证

**① 同质多智能体辩论**不如单体自我修正。
arXiv **2605.00914**《The Cost of Consensus: Isolated Self-Correction Prevails Over Unguided Homogeneous Multi-Agent Debate》**[已核实]**：失败模式是**谄媚顺从**（modal adoption 高达 85.5%），且辩论多烧 2.1–3.4× token 换来同等或更低准确率。

> 对场景 4 的直接约束：**评审者必须异质**。arXiv **2604.02923**《Council Mode》**[已核实]** 用异质前沿模型 + 显式区分共识/分歧/独家发现，幻觉率相对下降 41.7%，代价 4.2× token——**这个结构和我们的框架几乎一致**。

**② Anthropic 自己说主从式不适合编码。**
其多智能体研究系统博文明确写道：该架构**不适合**需要共享上下文、或依赖关系密集的任务，并**点名「most coding work」表现不佳**；且消耗约 **15× token**。

> 对场景 6 的直接约束：我们的场景恰恰是编码。**这条反证比那个 90.2% 更重要。**

**③ 90.2% 是厂商第一方数据**，非独立复现，测的是研究类任务，不是代码。

## 四、逐格详情

### 4.1 shell-out 通道

**互查 Review ✅**
- `sendbird/cc-plugin-codex` **[README 来源]**：反向——在 Codex 里 shell-out `claude -p`。小 diff 内联进 prompt，大 diff 退回让被调方自己读。
- `shimo4228/codex-review` **[已核实源码]**：bash 包 `codex review`，白名单参数（`--uncommitted`/`--base`/`--commit`），非白名单直接 exit 64 保证只读。
- santa-loop **[已验证本机]**：见第一节。
- **无一测量真阳性/重合率/成本。全是工作流自动化，没有度量。**

**并行评审 Consensus ✅**（本矩阵唯一「结构完全对上」的先例）
`AltimateAI/claude-consensus` **[已核实源码]**：
```bash
codex exec -s read-only {EXTRA_DIRS} -o $SESSION_DIR/{MODEL_ID}.md - < $SESSION_DIR/prompt.md
codex exec resume --last - < $SESSION_DIR/convergence-prompt-{MODEL_ID}.md
```
- push 上下文（stdin），`-o` 落盘，多模型 fan-out（codex / agy / qwen），再做收敛轮
- Claude 侧用 Task 工具的 teammate，**不是** `claude -p` 子进程 → 两条腿不对称
- 同时提供 `.claude-plugin/` 与 `.codex-plugin/`，双向
- **26 stars，无度量**

**互补 Consult ✅ —— 且是一条重要证据**
`stevekinney.com` 的 codex-advisor：Claude 在特定触发条件（不可逆架构决策、连续 2 次修复失败后的根因分析、认证/加密安全评审、低置信标记）下自主升级询问 Codex。

作者**明确弃用了 MCP 路径**，原文大意：MCP server 造成「运行盲区」——超时挂起会话、限流产生歧义失败；改为把 `codex exec` 包在带空闲超时看门狗的 shell 脚本里。

> **这是「MCP 适合 Consult」这一论断的直接反证，且来自真正实现过它的人。**

**互接 Handoff ◐**
`formin/multi-model-review`：Claude Code 写出 `review-package.md` 到磁盘，**人类在另一个终端**跑 Codex。是文件交接，但**不是一个 CLI 调另一个**。它是唯一报告了量化数字的项目，但测的是 token 压缩率，不是评审质量。

**主从式 ✅**
`boshu2/agentops` 的 Codex Team skill **[README 来源]**：Claude 做 lead，先做文件冲突分析（lock manager），再拆成并行/合并/多波次，spawn 多个 Codex agent 并汇总。走 CLI，不走 MCP。

**测试资产评审 ❌**
两路调研独立确认：找不到任何「两个 agent 独立评审测试用例/测试方案本身」的实现。

### 4.2 MCP 通道 —— 证据指向「不要用」

**没有任何公开的效果测量。** 全部是博文、README、issue。

`codex mcp-server` **[已验证本机]** 暴露且仅暴露两个工具：
```
codex        -- Run a Codex session
codex-reply  -- Continue a Codex conversation by providing the thread id and prompt
```

**已核实的四类问题：**

1. **上下文隔离被结构性破坏**：Codex 的完整响应（含推理文本）作为 `CallToolResult` 直接回流进 Claude 的主上下文。想真隔离，只能另起 `claude --mcp-config` 进程——那就等于放弃了 MCP 的便利。
2. **`codex-reply` 长期损坏**：`openai/codex` 至少 8 个 issue 跟踪 `conversationId`/`threadId` 不返回、重启后 "Session not found"（#3712 / #4651 / #5660 / #8388 / #8580 / #8726 / #12596 / #13641）。
3. **官方插件自身也挂**：`codex-plugin-cc` issue #258——插件用 `approvalPolicy:"never"` 却拒绝服务端发起的 elicitation，导致 `/codex:review` **无限挂起**。
4. **实践者主动弃用**（见上文 codex-advisor）。

**且 MCP 的唯一独占卖点（多轮会话）已被 `codex exec resume` 证伪。**

> **注意区分**：`codex mcp` = Codex 作为 MCP **客户端**（写入 `~/.codex/config.toml`）；`codex mcp-server` = Codex 作为**服务端**。OpenAI 的旗舰文档页 `developers.openai.com/codex/mcp` 讲的是**前者**，不是后者。唯一记录服务端接口的官方文件是 `codex-rs/docs/codex_mcp_interface.md`，且自陈 **experimental and subject to change**。

注册方式（社区来源，flag 已对照官方 CLI 文档）：
```bash
claude mcp add --scope user --transport stdio codex -- codex mcp-server
```

### 4.3 官方插件通道 —— 在本机结构性损坏

**命令 → 场景映射** **[已核实源码]**

| 命令 | 场景 | 备注 |
|---|---|---|
| `/codex:review` | 互查 Review | 调 `review/start` RPC，**无自定义 prompt** |
| `/codex:adversarial-review` | 互查（对抗变体） | 自有 prompt + `schemas/review-output.schema.json` 强制结构化 |
| `/codex:rescue` | 互补 Consult（委派） | 只转发任务文本，不传文件 |
| `/codex:transfer` | 互接 Handoff | `externalAgentConfig/import` 导入 Claude JSONL |
| `/codex:status` `/result` `/cancel` | 无场景 | 后台作业管道 |
| `/codex:setup` | 无场景 | 环境引导 |

**上下文传递是混合的，而这决定了它在本机的死活**：

- `/codex:review`：`review/start` 的入参只有 `{mode, baseRef, label}`，**从不传 diff 文本**。Codex 必须在自己的 `sandbox:"read-only"` turn 里读工作区。
- `/codex:adversarial-review`：**尺寸门控**——`DEFAULT_INLINE_DIFF_MAX_FILES = 2`、`DEFAULT_INLINE_DIFF_MAX_BYTES = 256KB`。小于阈值则把 `git diff` 文本内联进 prompt；**超过则提示 Codex「自己用只读 git 命令去看」**。
- `/codex:rescue`：从不传文件内容。
- `/codex:transfer`：传的是文件**路径**，且 import 走非沙箱的 app-server RPC。

**结合本机已验证的沙箱故障**（`read-only` 与 `workspace-write` 下 Codex 起子进程均报 `CreateProcessWithLogonW failed: 1385`，仅 `danger-full-access` 可读）：

| 插件命令 | 本机可用性 |
|---|---|
| `/codex:review` | ❌ **必然失败**（从不 push，100% 依赖 Codex 自读） |
| `/codex:adversarial-review` | ◐ **仅 ≤2 文件且 ≤256KB 的 diff 可用**；超出即退化为自读 → 失败 |
| `/codex:rescue` | ❌ 必然失败 |
| `/codex:transfer` | ◐ import 步骤本身不受影响；恢复后的 Codex 线程一旦干活仍会撞上 |

**插件结构上无法表达的东西**：并行评审 fan-out（无任何多线程代码路径）、评审者之间的重合率/一致性度量、成本核算。

**Stop-hook 评审门禁**确实存在 **[已核实源码]**：`hooks/hooks.json` 注册 `Stop` 钩子，解析 Codex 回答首行 `ALLOW:` / `BLOCK:`，`BLOCK` 时返回 `{decision:"block"}` 阻止 Claude 结束回合。默认 `stopReviewGate=false`。README 自陈会「drain usage limits quickly」。

## 五、这轮调研的净结论

**1. 三种交互方式并非在每个场景都语义等价。**
- 插件通道**无法表达**并行评审。
- MCP 通道会结构性破坏上下文隔离——而隔离是互查 Review 的前提（arXiv 2603.12123 已证隔离是收益来源）。**所以 MCP × 互查 Review 不是「另一个实现选项」，而是会破坏该场景前提的选项。**
- shell-out 是唯一在六个场景里都能表达的通道，且新证实支持多轮会话。

**2. 本机环境把插件通道大部分格子直接判死。** 这本身就是实验结果，不是障碍。

**3. 整个生态没有任何一个项目测量过效果。** 所有先例都是工作流自动化：没有真阳性率、没有重合率、没有成本核算。`formin/multi-model-review` 唯一的数字是 token 压缩率。

> **所以我们要测的东西，确实没人测过。** 但也因此，不能引用任何现成数字——包括那些 star 数很高的项目。

**4. 六个场景里有一个是原创的。** 测试资产评审在公开生态中找不到先例（两路独立调研 + 本机 `ecs-qa` 只有同模型实现）。它应当被标注为**本地假设**，靠类比场景 1/4 来论证，而非伪造引用。

**5. 本机已有两套可复用的同模型实现**（`ecs` santa-loop、`ecs-qa` 并行评审），但 `ecs-qa` 的评审 agent frontmatter 是
```yaml
tools: ["Read", "Write", "Edit", "Glob", "Grep", "Bash"]
```
**评审者带 Write/Edit 权限**，违反「评审者只读」这一不变量。改造前必须先收权限。

## 六、尚未回答 / 无法回答

- **MCP 通道的效果无人测量过**，我们若要填这一列，得自己定义怎么在「模型自主决定何时调用」的前提下做可复现实验。这是真难题，不是工作量问题。
