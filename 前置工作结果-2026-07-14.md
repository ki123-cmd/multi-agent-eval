## 5. 过程中修掉的两个真 bug（都会污染数据）

1. **`FileNotFoundError`**：Windows 上 `claude` 是 npm 的 `.cmd` 垫片，`subprocess` 不走 shell 找不到它。
   → **被记成了"CC 没做出来"**。已改为 `shutil.which` 解析真实可执行文件。
2. **"命令行太长"**：prompt 走 argv，Windows CreateProcess 上限 8191 字符，题面长的 case（如 `beer-song`）必炸。
   → 同样被记成任务失败。已改为**走 stdin**。

**据此加固了 harness**：`cc.ok == False`（基础设施故障）**不计入 pass rate 的分母**，且不写进"已完成"集合，
下次运行自动重试。这与官方 Codex 插件 Stop hook 的 fail-closed 缺陷（超时 / 非零退出 / JSON 解析失败
全都走 block 分支）正好相反——**基础设施故障和"模型判断有问题"必须在数据上分得开**。

## 6. Martian 数据的已知偏差（必须在论文里声明）

这些 PR 是 Martian **有意构造的合成 PR**（fork 到 `ai-code-review-evaluation` 组，在分支上注入缺陷）。
部分 PR 的 base 不是 master 而是另一个特性分支，甚至多个 PR **共用同一个 head 分支**。

例：`sentry-1`（审计日志分页，3 文件）与 `sentry-2`（spans buffer，9 文件）——**后者的 diff 完整包含前者的 3 个文件**。

- **本地 diff 是忠实的**：30/30 与 GitHub 自报的 `changed_files` 逐个对账一致（`cases/martian/_integrity.json`）。
- **但存在 6 组 diff 重叠的 PR 对**：reviewer 会对不属于本 PR 的代码提意见，而 golden comments 不覆盖它们
  → **会被误算成误报**。误报率统计时必须把重叠文件上的意见单独归类。

## 7. 建议：把 Martian 提为 Context Payload 主实验载体

**理由：**

1. **它直接测 reviewer 质量**（golden comment 召回率）——正是"喂什么上下文能让 reviewer 更强"这个研究问题的因变量。
2. **没有天花板效应**——review 召回率天然远低于 100%（Cross-Context Review 的 F1 才 28.6%）。
3. **不需要 CC 先写代码**——PR diff 就是现成产物，省掉一半流程和一半成本。
4. **30 个 diff 已拉好并校验完，今天就能跑。**
5. ⭐ **可以先用 CC 当 reviewer 跑起来，完全不依赖 Codex 账号**——而 KDD'26 那篇恰恰说"Claude 审"是占优方向。
   等 Codex 恢复后再加 Codex-as-reviewer 那条臂做跨模型对比。

**P0..P6 在 Martian 上的映射：**

| 代号 | 交接包 | 对应现实 |
|---|---|---|
| `P0` | 只给 PR 标题/描述，**不给 diff** | **官方插件 Stop hook 的等价物**（劣势基线） |
| `P1` | 只给 diff | 最朴素 |
| `P2` | diff + PR 描述 | 常见做法 |
| `P3` | diff + 被改文件的完整内容 | 加厚上下文 |
| `P4` | diff + 描述 + 仓库结构 / 相关调用方 | 带环境 |
| `P5` | P4 + "重点关注 X" 的引导 | 有先验 |
| `P6` | 不给 diff，只给改后的完整文件 | 去锚定 |

**因变量**：golden comment 召回率（按 severity 分层，Critical 单列）+ 误报率（扣除 6 组重叠偏差）。

**Polyglot / BugsInPy 退居"端到端增益"验证**，等 WSL/Docker 就位再做。
