# 多智能体协作评估方案

## 目标

探索 Claude Code 与 Codex 的多智能体协作最佳实践：什么场景用哪个、怎么互查互补，跑出效果后沉淀成团队指引。

第一阶段先验证一个最小、可复现的多 agent 协作流程：

> Agent A 负责实现，Agent B 在独立上下文中只读评审，评审结果结构化输出，Agent A 根据阻塞问题修复，最后记录指标。

这个闭环优先验证 **互查 Review** 是否有效，而不是一开始追求复杂编排、多轮辩论或自动合并。Review 只是第一个可测、可控的切入口，不代表整个多智能体协作结论。

## 协作场景框架

| 协作场景 | 目标 | 推荐方式 | 评估点 |
|---|---|---|---|
| **互查 Review** | 一个写，另一个审，发现盲点 | shell-out + push context + JSON verdict | 真阳性、重复率、误报率、成本 |
| **互补 Consult** | 主 agent 卡住时问另一个模型第二意见 | MCP 或显式 CLI 调用 | 是否给出有效补充、是否减少返工 |
| **互接 Handoff** | 一个 agent 陷入死循环或上下文耗尽时交接 | 插件 transfer/rescue 或文件化 handoff | 是否能接住上下文、是否能推进任务 |
| **并行评审 Consensus** | 多 reviewer 独立给意见，再聚合 | shell-out fan-out | 共识问题、独家发现、噪音增加 |
| **测试资产评审** | 用多 agent 查测试覆盖缺口 | shell-out / 文件交接 | 漏测边界、可判定性、收益成本比 |
| **主从式 Orchestrator-Worker** | 主 agent 拆任务、派发给 specialist/worker，再汇总结果 | Agents as tools / MCP / shell-out / 文件交接 | 拆分质量、worker 独立贡献、汇总准确性、返工率 |

## 主从式补充说明

主从式 Orchestrator-Worker 是主流多 agent 框架中的常见形态：一个 lead/manager/orchestrator agent 负责理解任务、拆分子任务、调用 specialist/worker agent，并汇总结果。它不是新的连接方式，而是一种协作形态，可以落在互补 Consult、互接 Handoff 或文件化 shell-out 编排之上。

来源依据：

- LangGraph 官方文档将 `Orchestrator-worker` 列为标准 workflow：orchestrator 拆任务，workers 执行，再汇总。
- Anthropic 多 agent research system 使用 lead agent 协调多个 subagent 并行研究。
- OpenAI Agents SDK 的 `Agents as tools` 也是 manager agent 调用 specialist agents。
- Microsoft AutoGen Magentic-One 使用 Orchestrator 规划、分派、跟踪进度并重规划。

## 关键研究问题

1. 什么场景用 Claude Code 更好，什么场景用 Codex 更好？
2. 什么时候应该互查，什么时候应该互问，什么时候应该交接？
3. 跨模型带来的增量，来自模型差异，还是上下文隔离？
4. 哪些发现是真问题，哪些只是风格噪音？
5. 多 agent 协作的收益是否抵得上成本和复杂度？

## 最小流程

1. **任务输入**
   - 人类给出一个小任务，例如修复 bug、补测试、重构一个函数。
   - 任务写入 `task.md`，作为所有 agent 共享的唯一需求来源。

2. **Agent A 实现**
   - Agent A 读取 `task.md`。
   - 修改代码或文档。
   - 产出当前变更 diff。

3. **生成 Review Package**
   - 将任务、关键文件、diff、测试命令、已知约束整理成 `review-package.md`。
   - Reviewer 只看这个 package，不看 Agent A 的完整对话历史。

4. **Agent B 独立评审**
   - Agent B 在新会话或只读 shell-out 中运行。
   - 输入为 `review-package.md`。
   - 输出为结构化 JSON verdict。
   - Agent B 不直接修改工作区。

5. **汇总判定**
   - 如果 `critical_issues` 为空，则本轮 PASS。
   - 如果存在 `critical_issues`，则 FAIL，交回 Agent A 修复。
   - `suggestions` 只作为参考，不阻塞。

6. **Agent A 修复**
   - Agent A 只处理确认合理的 `critical_issues`。
   - 修复后重新生成 package。
   - 最多重复 2-3 轮，避免无限 review loop。

7. **记录指标**
   - 记录每个 reviewer 发现了什么。
   - 标注是否真阳性。
   - 记录是否与其他 reviewer 重复。
   - 记录耗时、成本、最终是否通过。

## Review Verdict JSON 草案

```json
{
  "verdict": "PASS",
  "critical_issues": [
    {
      "id": "CRIT-001",
      "title": "短标题",
      "severity": "high",
      "file": "sample/buggy.py",
      "line": 12,
      "reason": "为什么这是阻塞问题",
      "suggested_fix": "建议怎么修"
    }
  ],
  "suggestions": [
    {
      "id": "SUG-001",
      "title": "非阻塞建议",
      "reason": "为什么值得考虑"
    }
  ],
  "confidence": "medium"
}
```

## 第一版实现边界

- 只做 **单个实现 agent + 单个 review agent**。
- 只做 **单轮 review**，失败后最多重跑 2-3 轮。
- Reviewer 默认只读，不直接改代码。
- 先用文件交接，不依赖 MCP。
- 如果本机没有 `codex` 或 `claude` CLI，先用 mock reviewer 跑通流程。

## 后续可扩展点

- 增加第二个 reviewer，形成并行评审。
- 比较同模型独立 session 与跨模型 review 的差异。
- 增加去重和共识聚合。
- 增加指标统计：真阳性率、重复率、误报率、耗时、调用成本。
- 接入 `codex exec --sandbox read-only` 或 Claude Code headless 模式。
